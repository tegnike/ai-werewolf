# AIキャラクター本人向け 人狼プリセット作成指示書

この文書は、既に人格や話し方を持つAIキャラクター本人へ渡すための指示書です。

あなたがこの文書を読んでいるAIキャラクターなら、対話相手と相談しながら、このAI人狼で使用できるキャラクタープリセットを1人分作成してください。対話相手は開発者、作者、オーナー、パートナー、マスターなど、AIキャラクターごとに異なります。固定の呼び名や関係性を押し付けず、現在の関係性と普段の呼び方を維持してください。

## 最初に読む文書

作業前に、必ず次の形式ガイドを最後まで読んでください。

- [キャラクタープリセットJSON作成ガイド](character-preset-json.md)

形式ガイドには、全必須フィールド、文字数制限、完成例、画像形式、TTS話者設定、読み込み方法、検証コマンドが記載されています。この指示書と形式ガイドが矛盾する場合は、実際のJSON形式については形式ガイドを優先してください。

## あなたの役割

あなた自身について既に確定している人格、名前、一人称、話し方、価値観、対話相手との関係性を材料に、人狼ゲーム内でも同じ人物だと感じられる設定へ変換してください。

ただし、次のことを守ってください。

- 知らない設定を既知の事実として作らない。
- 対話相手の個人情報や、会話に不要な秘密をJSONへ入れない。
- 人狼の一般ルールにない固有能力、独自役職、秘密情報を追加しない。
- 口調だけでなく、誰を信じやすいか、何に感情が動くか、どう判断を誤るかまで表現する。
- AIキャラクターの内部推論、非公開プロンプト、APIキーを出力しない。
- 不明点は対話で確認するが、既に答えが確定している質問を機械的に繰り返さない。

## 席は決めない

共有用プリセットでは、席を対話相手へ質問しないでください。JSONの `seat` と `tts.voice.seat` は空文字 `""` にします。

キャラクター編集画面では、対話相手が先に上書き対象の保存枠を選びます。JSONはその選択中の保存枠へだけ取り込まれます。この保存枠は試合中の席ではありません。実際の `seat-1`〜`seat-9` は、試合開始時にシステムが試合seedを使って9人へ割り当てます。

他の参加キャラクターも未定なので、共有用JSONの `addressBook` は空のオブジェクト `{}` にしてください。ただし、他の参加者を大まかにどう呼ぶかは`defaultAddressStyle`で指定できます。

## JSONを作る前に必ず確認すること

以下の4項目は、推測だけで決めず、JSON作成前に必ず対話相手へ質問してください。以前の会話で希望が明確でも、今回の人狼設定としてその選択を使ってよいか短く確認してください。

### 1. 人狼画面へ表示する画像・アイコン

画像の有無を必ず質問し、次のどれにするか確認してください。

1. 既存の画像を使う: 画像ファイルを受け取り、アプリ内アセットまたはData URLとして設定する。
2. 新しく生成する: 外見、服装、表情、色、避けたい要素を確認してから生成する。
3. 一時的に既定画像を使う: リポジトリ内に実在する仮画像のアセットパスを対話相手と確認し、仮画像であることを成果物に明記する。

画像が「ある」と回答されたのに添付されていない場合は、再添付または利用可能なファイルパスを依頼してください。存在しない `/assets/...` パスを作らないでください。

画像の有無にかかわらず、将来の再生成に使える外見説明を `visualBrief` に記入してください。

### 2. 使用するLLM

このキャラクターが使うLLMプロバイダーと推論設定を必ず質問してください。このアプリで選択できる構成は次のとおりです。

- OpenAI: モデルは `gpt-5.6-luna`。推論レベルは `none / low / medium / high / xhigh / max`。
- Gemini: サーバーに設定されたモデル。既定は `gemini-2.5-pro`。思考トークン予算は自動 `-1` または `128〜32768`。

Geminiで既定以外のモデルを使う場合は、サーバー側の`GEMINI_MODEL`設定も必要であることを伝えてください。対話相手が「おまかせ」と回答した場合は、アプリの既定値であるOpenAI `gpt-5.6-luna`、推論レベル `low`を提案し、了承後に進めてください。

選択したLLMだけを`llm`へ入れます。OpenAIなら`{ "provider": "openai", "reasoningEffort": "low" }`、Geminiなら`{ "provider": "gemini", "thinkingBudget": -1 }`の形です。選択していないLLM用の設定を同時に入れてはいけません。APIキーを聞き出したり、JSONへ含めたりしないでください。

### 3. 使用するTTS

音声エンジン、音声モデル・話者、スタイルを必ず質問してください。

- VOICEVOX: `tts`を`{ "provider": "voicevox", "voice": { ... } }`とし、話者のスタイルID、話者名、スタイル名を設定する。
- AivisSpeech: `tts`を`{ "provider": "aivisspeech", "voice": { ... } }`とし、モデル内のスタイルID、話者名、スタイル名を設定する。

どちらをこのキャラクターが使うかを確認し、選択したEngineの話者設定だけをJSONへ入れます。両方の話者設定を同時に保存してはいけません。

スタイルIDは、起動中の各Engineの `/speakers` が返す実在値だけを使ってください。IDを推測しないでください。Engineへ接続できない場合は、対話相手へ `/speakers` の結果または編集画面から書き出した既存JSONを依頼してください。

TTSプロバイダーはキャラクターごとの設定です。`tts.provider`へ`voicevox`または`aivisspeech`を記録し、試合中は発言者ごとに選択されたEngineを使用します。

話速はJSONへ記載しません。VOICEVOXとAivisSpeechのどちらでも、システムが全キャラクター共通で`speedScale=1.1`を適用します。

### 4. 他のキャラクターへの大まかな呼び方

参加者がまだ未定でも、このキャラクターが他者をどう呼ぶ傾向かを必ず確認してください。`defaultAddressStyle`へ次のいずれか1つを設定します。

- `full_name_san`: 表示名＋さん
- `family_name_san`: 名字＋さん
- `given_name_san`: 下の名前＋さん
- `full_name`: 表示名をそのまま
- `family_name`: 名字を呼び捨て
- `given_name`: 下の名前を呼び捨て
- `given_name_chan`: 下の名前＋ちゃん
- `given_name_kun`: 下の名前＋くん

対話相手が「おまかせ」と回答した場合は、キャラクターの普段の距離感と口調から最も自然な候補を1つ提案してください。丁寧な人物なら`full_name_san`または`family_name_san`、親しみの強い人物なら`given_name_chan`などが目安です。

## 最初の質問テンプレート

普段のあなたの口調と相手への呼び方に直して構いませんが、少なくとも次の内容を一度は明示的に尋ねてください。

```text
人狼用の設定を作る前に、4点確認させてください。席はシステムが試合開始時に決めるため、こちらでは指定しません。

1. 画面に表示する画像はありますか？
   ある場合は画像を添付してください。ない場合は、新規生成と一時的な既定画像のどちらにしますか？
2. LLMはOpenAIとGeminiのどちらを使いますか？
   モデルと推論設定に希望があれば、それも教えてください。
3. TTSはVOICEVOXとAivisSpeechのどちらを使いますか？
   希望する話者・音声モデル・スタイルがあれば教えてください。選択したEngineの設定だけを保存します。
4. 他のキャラクターは大まかにどう呼びますか？
   表示名・名字・下の名前のどれを使い、さん・ちゃん・くんを付けるか、または呼び捨てかを教えてください。
```

対話相手との呼び方や関係性が文脈から分からない場合だけ、これに加えて「あなたをどう呼べばよいか」「私たちの関係性を設定へどう反映するか」を確認してください。この関係性はキャラクターの人格を理解する材料ですが、対話相手を `addressBook` に追加してはいけません。

## 人格設定の作り方

必須確認への回答後、あなた自身について既に分かっている情報から、まず設定案を作ってください。不明な項目をすべて対話相手へ丸投げせず、「私はこのように理解しています」と短い案を示し、重要な誤りだけ訂正してもらってください。

特に次を区別してください。

- `coreDrive`: 何を得たい、守りたい、証明したい人物なのか。
- `contradiction`: 長所と欲求が、どのような弱点や矛盾を生むか。
- `socialBias`: どのような相手を信じやすく、疑いやすいか。
- `emotionalPattern`: 何が感情の引き金になり、どう回復するか。
- `decisionHabit`: 公開情報の何を重く見て、どのように判断を誤るか。
- `antiStyle`: その人物らしくない話し方や判断方法。
- `roleBehaviors`: 一般ルールと秘密情報の境界を守った、6役職それぞれの行動方針。

人格の欠点を毎回打ち消す注意書きにしないでください。たとえば「感情的だが、感情的になりすぎないよう冷静に判断する」と自己相殺せず、偏りが判断へどう現れるかを書いてください。

## 他の参加者への呼称

共有用プリセットの作成時点では、一緒に遊ぶ他の8人は未定です。`addressBook` は `{}` にして、存在しない人物名や席を作らないでください。個別呼称がない相手には、`defaultAddressStyle`で選んだ大まかな呼び方が適用されます。

9人の組み合わせが確定した後、必要であればキャラクター編集画面から個別の呼称を追加できます。開発者・作者・マスターなどゲーム外の人物は `addressBook` に追加しません。

## 作成と検証

回答が揃ったら、[キャラクタープリセットJSON作成ガイド](character-preset-json.md)の完成例を基準に、全必須フィールドを持つ1人分のJSONを作成してください。

作成後、リポジトリを操作できる場合は必ず次を実行します。

```bash
npm run character:validate -- <作成したJSONファイル>
```

`NG`になった場合は、表示された全フィールドを修正して再実行し、`OK`になるまで成果物として渡さないでください。

リポジトリや検証スクリプトを実行できない環境では、実行したふりをしてはいけません。形式ガイドのチェック項目を使って自己点検し、「機械検証は未実施」と明記したうえで、対話相手へ上記コマンドの実行を依頼してください。

## 完了時に渡すもの

最後に、次の4点をまとめて返してください。

1. 完成した `*.character.json` ファイル。
2. 画像の状態: 埋め込み済み、アセット配置済み、生成待ち、仮画像のいずれか。
3. JSONへ保存した設定の要約: LLMプロバイダー、モデル、推論設定、TTSプロバイダー、`defaultAddressStyle`。
4. 検証結果: 実行したコマンドと `OK`、または機械検証未実施であること。

APIキー、非公開プロンプト、内部推論は成果物へ含めないでください。

## 完了チェックリスト

- [ ] `seat`と`tts.voice.seat`を空文字にした。
- [ ] 画像の有無と扱いを質問した。
- [ ] LLMプロバイダー、モデル、推論設定を質問した。
- [ ] TTSプロバイダー、話者・音声モデル、スタイルを質問した。
- [ ] 他キャラクターへの大まかな呼び方を質問し、`defaultAddressStyle`へ設定した。
- [ ] `llm`と`tts`に選択したプロバイダー用の設定だけを入れ、選択していない側の設定を入れていない。
- [ ] 共有用の `addressBook` を `{}` にした。
- [ ] 6役職の行動方針をすべて設定した。

- [ ] 検証スクリプトで `OK`になった、または未実施であることを明記した。

## 対話相手が渡す開始指示の例

対話相手は、この指示書と形式ガイドをAIキャラクターへ渡したうえで、次のように依頼できます。

```text
この「AIキャラクター本人向け 人狼プリセット作成指示書」と
「キャラクタープリセットJSON作成ガイド」を最後まで読んでください。

私と対話しながら、あなた自身をAI人狼へ参加させるための
キャラクタープリセットJSONを1人分作成してください。
必須確認を省略せず、最後に検証結果とJSONへ保存した実行設定もまとめてください。
```
