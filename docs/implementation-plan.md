## 0. 最重要方針

これは**一般的な人狼ゲームをAI同士にプレイさせるアプリ**である。独自ゲームへ改変しない。

- ゲーム名は画面上もREADME上も、単に `AI人狼` / `AI Werewolf` とする。
- 役職名は一般的な `村人・人狼・占い師・霊媒師・狩人・狂人` をそのまま使う。
- 人狼、村人、占い師などを別の固有名詞へ言い換えない。
- 独自の世界観、物語、職業設定は付けない。プレイヤー人格は一般的な人間関係の範囲で編集できる。
- AIは9人の個別人格を持つプレイヤーとして参加する。ただし独自役職・物語・世界観は付けない。
- 一般的な9人人狼のルールを採用し、ルール文、コード、画像、UIは独自制作する。

本書は、Fableが提案した観戦UXと堅牢な技術構成を、マスターの訂正に従って**標準人狼へ全面的に戻した修正版**である。以前の独自タイトル・独自役職・独自世界観はすべて破棄し、実装へ持ち込まない。

## 1. プロダクト

### 1.1 名称と目的

- 日本語名: `AI人狼`
- 英語名: `AI Werewolf`
- リポジトリ: `ai-werewolf`

> 9体のAIエージェントが、一般的な人狼ゲームを最初から最後まで自律プレイし、人間が公開視点またはGM視点で観戦できるWebアプリ。

### 1.2 MVP

- AIプレイヤーは9人固定。編集用の保存枠と試合中の席を分離し、試合開始時にseedで9人を決定論的に配置する。
- 初期9名は名前、行動原理、欠点、感情傾向、話し方、役職別行動方針、呼称、立ち絵、音声を持つ。
- 新規試合は開始時の9人分の設定を不変スナップショットとして保存し、後からプリセットを編集しても進行中の試合と過去のリプレイは変化しない。
- アプリがGMを担当する。
- 人間は開始、観戦、一時停止、再開、中断、リプレイのみを行う。
- 昼の議論、投票、夜の人狼会話・襲撃・役職行動を完全自動進行する。
- 公開視点とGM視点を切り替えられる。
- SQLiteへ全イベントを保存し、試合後にリプレイできる。
- ローカル／セルフホスト前提。Docker対応。
- 観戦UIはPCのデスクトップブラウザ専用とする。
- 画像は実装セッションがCodex組み込み`imagegen`で生成する。
- ON/OFF・個別音量調整可能なオリジナルBGMと、VOICEVOXまたはAivisSpeechによる9名別々の読み上げを備える。
- VOICEVOXとAivisSpeechの話速は全キャラクター共通で `speedScale=1.1` とする。AivisSpeechの互換APIから返されたAudioQueryも、他のEngine固有値は維持したまま`speedScale`だけを1.1へ上書きする。
- 「人狼」と人名の漢字表記はVOICEVOX／AivisSpeechのユーザー辞書APIへ固有名詞として同期し、Engineや話者が変わっても読みを統一する。同期はEngineごとに冪等とし、アプリ外の辞書項目を変更しない。

### 1.3 非ゴール

- 人間プレイヤー参加。
- 役職・人数のカスタマイズ。
- アカウント、課金、ランキング、マッチメイキング。
- 多言語対応。
- スマートフォン向けデザイン、タッチ操作最適化、モバイル用レイアウト。
- サーバーレス、複数サーバー、水平スケール。
- 特定の既存人狼製品との互換性・再現。

## 3. 標準9人ルール

### 3.1 配役

| 役職 | 陣営 | 人数 | 能力 |
|---|---|---:|---|
| 村人 | 村人陣営 | 3 | 特殊能力なし。会話と投票で人狼を探す |
| 人狼 | 人狼陣営 | 2 | 仲間を知り、夜に相談して1名を襲撃する |
| 占い師 | 村人陣営 | 1 | 毎夜1名が人狼か否かを知る |
| 霊媒師 | 村人陣営 | 1 | 前日に処刑された人物が人狼か否かを知る |
| 狩人 | 村人陣営 | 1 | 毎夜1名を人狼の襲撃から守る |
| 狂人 | 人狼陣営 | 1 | 人間だが、人狼陣営の勝利を目指す。特殊能力なし |

重要:

- 人狼2名は互いを知る。
- 人狼は狂人が誰か知らない。
- 狂人は人狼が誰か知らない。
- 占い師が狂人を占うと `人狼ではない` と出る。
- 霊媒師が処刑された狂人を判定しても `人狼ではない` と出る。
- 狂人は人数判定上は非人狼として数えるが、人狼陣営が勝てば勝利する。
- 狂人自身が死亡していても、人狼陣営が勝てば狂人の勝利とする。

### 3.2 勝利条件

- 村人陣営勝利: 生存する人狼が0名になる。
- 人狼陣営勝利: 生存人狼数が、生存する非人狼人数以上になる。
- 狂人は勝敗判定の「人狼数」には含めない。
- 判定は夜明け後と処刑後に行う。

例:

- 人狼1＋村人系1: 人狼勝利。
- 人狼1＋狂人1: 人狼1、非人狼1なので人狼勝利。
- 人狼1＋狂人1＋村人1: 人狼1、非人狼2なので継続。
- 人狼0＋狂人1: 村人勝利。

### 3.3 第0夜

初日の議論材料を残すため、第0夜は襲撃なしとする。

1. GMがゲーム開始を宣言する。
2. 人狼2名に仲間を非公開通知する。
3. 人狼2名が1回ずつ非公開で顔合わせ会話をする。襲撃先は選ばない。
4. 占い師が自分以外の1名を選んで占う。
5. 占い結果を占い師本人だけへ通知する。
6. 霊媒師と狩人は行動しない。
7. 第1日へ進む。

### 3.4 昼

1. 2日目以降は、前夜の犠牲者名、または犠牲者なしをGMが公開する。襲撃のない第0夜明けは省略し、1日目は最初の発言から始める。
2. 勝敗判定。
3. 固定順を設けず、動的な話者選択による昼の議論を行う。生存者全員に最低1回の発言機会を保証する。
4. 生存者全員が同時投票する。
5. 記名投票結果を公開する。
6. 最多得票者を処刑する。
7. 処刑者の名前だけ公開し、役職は試合終了まで公開しない。
8. 勝敗判定。
9. 未決着なら夜へ進む。

### 3.5 発言

- 昼の議論は固定の開始発言一巡を持たず、最初から最後まで1本の動的な発言キューで進行する。最初の話者と、指名・必須主張・発言希望がない場合の次話者はseedから決定論的に選ぶ。
- 同じ人物の発言間には必ず別の2人の発言を挟み、`A → B → C → A` より短い間隔で再発言させない。返答要求、必須の役職主張・対抗、集中先の弁明もこの2発言クールダウンを飛び越えず、対象が再発言可能になるまで別の話者を選ぶ。
- 発言者が特定の生存者へ返答を求めた場合は、構造化結果の `addressedTo` と `requestsReply` を正本とし、発言上限と2発言クールダウンの範囲でその相手を次の話者にする。台詞本文の文字列解析で指名を判定しない。
- 次話者の優先順位は、再発言可能な人の中で、返答を求められた人、必須の役職主張・対抗がある人、未発言者、発言希望者、seedによる選択の順とする。
- その日の先頭発言者は、まだ起きていない当日の沈黙、反応、便乗、返答遅延を観察したように話さない。公開済みの能力結果、自分がその発言で公開する能力結果、前日までの発言などの根拠がない場合は、疑い先・投票先・暫定評価を無理に作らず、確認したい論点、質問、今後の判断基準から自然に議論を始める。全話者は仮定・方針・弱い違和感を、具体的な告発へ勝手に強めない。
- 1日目turn 6までは、独立した2人以上の疑い、または結果つきの公開役職主張がまだない区間を「公開材料が少ない序盤」とする。この区間では、宛先つき質問、判断基準、役職の登場時機への意見、再判断条件つき保留を、それだけで完結した貢献として扱い、疑い先・処刑先を要求しない。公開材料が増えた後も候補指名を機械的に強制せず、複数の発言・役職情報を比べて差がなければ保留条件を示せる。turn 10以降は公開候補と本人の応答を比較して投票方針を示す区間とする。
- 生存者全員が最低1回話し、通常は1人1日最大2回とする。3人以上の投票予定が集まった対象本人だけは、合意形成後の反論として1回追加できる。9人生存時は通常発言14回未満で終了せず、通常枠18回に追加反論枠1回を加えた最大19回で必ず終了する。
- 8人以下では最低発言数を `max(生存者数, ceil(14 × 生存者数 / 9))`、最大発言数を `min(18, 生存者数 × 2)` とし、人数に比例して議論量を縮小する。
- 発言結果は本文に加えて、実際に話を向けた相手と返答要求の有無を返す。最後の発言枠では新しい返答要求を禁止する。
- 最低発言数と全員1回を満たした後に発言の連鎖が止まったとき、1日最大2回、各回最大4人へ非公開の発言希望確認を行う。発言希望は強さ、動機、対象だけをstrict schemaで返し、台詞や長い思考過程を含めない。
- 発言希望確認は、台詞生成用の話し方・呼称・台詞例を省いた短いプロンプトと、直近30件の公開履歴だけを使い、入力トークンを抑える。
- 発言希望の候補は、まだ確認されていない人、発言回数が少ない人を優先し、同点はseedから決定論的に選ぶ。
- 最低発言数と全員1回を満たしたうえで発言希望がない、2回の確認を使い切る、または最大発言数へ達すると投票へ進む。
- 1発言は日本語200 Unicode code point以内。
- 空文字は `……（沈黙）` として扱う。
- 役職を名乗ること、占い結果・霊媒結果の共有、騙り、嘘、弁明、質問は自由。
- システムは役職の名乗りや結果報告の真偽を証明しない。
- 新規試合の `discussion v3` では、各発言が本文と一致する `primaryAct`、質問または回答の話題、暫定的な疑い先・根拠分類・根拠日 `evidenceDay`、投票予定を構造化して返す。勘だけなら `evidenceDay=null` とし、公開情報を根拠にする場合はその情報の日を指定する。今日まだ未発言の人物でも前日以前の公開情報は根拠にできるが、当日の未観測の態度は根拠にできない。これは発言者の公開した立場を記録するもので、真の役職や内部推論ではない。
- 疑いの根拠分類では、一般的な発言内容に加えて、対象自身の言い間違い・言い回し `statement_slip` と、説明・理由の薄さや飛躍 `reasoning_quality` を区別する。既出の指摘へ明示的に同調した発言は公開された引用元を `echoSourceSeat` へ記録する。引用元が同じ対象への先行疑いまたは公開能力結果を持たない場合は、本文を再解析したり再試行したりせず `null` へ安全側に降格する。旧保存応答でこの値が欠けていても復旧を妨げない。
- エンジンはv3の構造化発言だけから、既出質問と回答数、疑いの分布、投票予定、盤面整理などの貢献数を日ごとに畳み込む。本文の文字列解析や別のAI呼び出しは使わない。同じ質問分類が2回明示された後はその分類を閉じ、後続の構造化出力では新たな返答要求に使えないようにする。回答として話題へ触れることは妨げない。
- エンジンは同じ構造化発言から、席ごとの最新の疑い候補と根拠分類・投票予定・回答済み分類・弁明済み・候補更新済みも日ごとに導出する。後続話者へこの最新公開立場と未発言者一覧を渡し、発言済みの人物を「未発言」「発言がない」「発言を聞けていない」と扱わせず、候補・根拠・回答・弁明が記録済みの人物について「まだ候補を出していない」「根拠をまだ出していない」「まだ答えていない」「まだ説明していない」という不存在主張をさせない。自分の発言への批判・質問と、自分が投票先にされていることを区別し、最新の疑い先・投票予定と矛盾する立場を他者へ帰属させない。当日の初発言で他者の質問・反応要求・疑いなどを自分の過去行動へ誤帰属する発言と、未公表の投票予定を「変えない」「維持する」と扱う発言は契約違反として同一モデルへ修正させる。内容の弱さ、質問とのずれ、理由への不同意は具体的に評価できる。追加AI呼び出し、秘密情報、自由本文から議論台帳を作る解析は使わない。
- 後続話者には、候補ごとの投票予定人数だけでなく、疑いを出したユニーク人数、根拠分類ごとのユニーク人数、独立した根拠分類数、既出論点へ同調した人数、公開された占い・霊媒結果と同役職主張人数をまとめた証拠台帳と、不足している貢献候補を渡す。発言を根拠にした暫定評価の候補には、当日すでに発言した人物だけを含める。投票予定の人数自体は証拠として扱わせず、同じ質問の反復より、独自の灰評価、弁明、役職候補の扱い、処刑方針を優先させる。候補は台詞や結論を固定する台本にはしない。
- `result` 以外の同じ対象・根拠分類を3人以上が述べたか、既出論点への同調が2人以上になった場合は、その論点を公開上の飽和論点として示す。後続発言では賛同を禁止せず短く述べることを許しつつ、別の公開材料、2番手候補、本人の応答との比較を優先させる。同じ相手を新しい根拠で疑うことや、人狼・狂人が有効な論点へ戦略的に便乗することは妨げない。反対意見・少数派・票割れを義務化しない。
- 同じ相手へ3人以上の投票予定が集まった後は、後続話者が同じ相手への投票予定を本文で追加宣言したり `voteIntent` へ設定したりしない。これは最終の非公開投票先を拘束せず、合理的な票合わせは妨げない。後続発言は増えた公開情報、被疑者への質問、反証、または未検討人物の検討へ使う。3人未満の段階で同一話者が投票先を変えず再宣言した場合は、本文と投票予定を保存するが `primaryAct` を `agreement` へ格下げし、主要な新規貢献として数えない。格下げは再試行を発生させず、`contributionDemoted=true` として記録する。
- 一般的な進行知識として、別々の対象への `人狼ではない` という結果は互いに矛盾しないこと、`人狼ではない` という結果だけの占い師候補2人を即座に処刑対象へ限定しないこと、`人狼` という結果が出た場合は対象本人と占い師候補を比較すること、霊媒師候補2人では順番に処刑する進行を検討することをv3へ与える。
- 占い師を名乗る人物が誰かへ `人狼` という結果を出しても、その主張だけで対象を確定人狼として扱わない。全員へ主張者と対象本人を比較させる。対象が人狼なら、その仲間には露骨な庇い一辺倒ではなく、占い師候補の信用低下、別候補の提示、処刑方針の再構成のいずれかで結論の固定を崩させる。狂人には人狼位置を与えず、結果の真偽を断定しないまま対抗主張、占い師候補への疑義、別候補の提示によって村の合意を揺らす機会として扱わせる。
- 村側では材料不足時の質問・判断基準・役職確認・条件つき保留を自然な貢献として扱う。人狼は公開情報だけで村人らしく通る説明を保ち、人格と盤面に応じて、迷いの提示、正当な疑問への誘導混入、代替仮説、進行案への誤候補混入、人狼仲間との距離調整、認可済みの役職騙りから任意に選ぶ。狂人は人狼位置を知らないまま、候補競合、判断基準の攪乱、役職候補への疑義、認可済みの対抗主張から任意に選ぶ。ブラフや候補捏造を毎発言の義務にしない。
- キャラクターの台詞では、能力結果や人物評価を「白」「黒」「白結果」「黒結果」「初手黒」などと省略しない。`人狼` / `人狼ではない` という構造化値は維持し、本文では「人狼だという結果」「人狼ではないという結果」のような自然な日本語で伝える。保存済みの公開発言に略語が含まれていても後続話者は模倣しない。
- 各人格は明示された一人称を使い、自分自身を自分の名前や敬称付きで呼ばない。役職主張の必須文も人格ごとの一人称を使う。自分の姓名・姓・名に「さん」「ちゃん」「くん」「君」「様」を付けた自己参照や、別の一人称への切替は発言契約違反として同一モデルへ修正させる。
- 2人以上から疑いを向けられたか、3人以上から投票予定を向けられ、当日まだ1回しか話していない人物には、必須の役職主張より後、未発言者のseed選択より前に1回だけ弁明機会を与える。まだ一度も話していない集中先は、通常の未発言者より先に発言させる。明示的な返答要求による次話者指定はこれより優先する。
- 新規試合の第0夜は公開発言・投票・能力情報が存在しないため、真占い師の合法対象をengineがseedと専用purpose key `d0-seer` から一様・決定論的に選び、AIのtarget判断を呼ばない。真偽どちらの占い師にも、第0夜の対象には推理上の選定理由が存在せず、事前の言葉の癖・反応・様子見・人格傾向を理由として発明してはならないと伝える。理由を尋ねられた場合は無情報選択だったと率直に答える。
- 全参加者は第0夜の占い対象選定を質問・攻撃・信用比較の材料にしない。1日目の議論promptとagendaは、材料不足時の質問を役職を名乗る時期、処刑方針、公開後の発言・反応へ限定し、第0夜の対象理由を尋ねたり評価したりする発言は発言契約違反として同一モデルへ修正させる。「結果の根拠」「なぜその人を人狼だと見たか」も、第0夜の対象選定理由を別表現で求める質問として同じ契約で拒否する。一人が無理由だと明言し、別の人が言及しなかったという差も信用比較に使わない。占い師候補は、名乗った時期、主張した結果、対抗との構図、結果を受けた本人の公開反応、名乗った後の一貫性で比較する。人狼・狂人の騙りも同じ公開規則に従い、ブラフは結果、登場時機、公開後の反応、代替仮説、人格に沿う言い回しへ表す。
- 占い師本人の第0夜履歴にはモデル生成理由ではなく「公開情報がないため推理上の選定理由は存在しない」という固定事実を渡す。1日目夜以降は公開材料があるため、対象と結果に加えて選択時の短い理由を従来どおり非公開履歴へ保持する。理由は公開イベントや他の参加者へ射影しない。

#### 3.5.1 役職主張・対抗CO・騙り

- 公開発言は、台詞とは別に任意の構造化主張を持てる。主張できる役職は占い師・霊媒師で、対象、対象夜／処刑日、本人が主張する `人狼` / `人狼ではない` の結果を含む。
- 構造化主張は真役職や真結果ではなく、その発言者が公に述べた内容である。真役職から自動生成せず、真占い師・真霊媒師・狂人・人狼を同じ発言契約と検証経路へ通す。
- 新規試合の `claims v4` では、真役職・狂人・人狼が、試合開始時にスナップショット保存されたキャラクター固有の役職主張設定と現在の公開盤面を読み、通常の昼発言と同じAI応答内で「今名乗る」「公開上の条件を指定して待つ」「この試合では潜伏する」を非公開の構造化 `claimIntent` として選ぶ。選択は確率抽選や座席番号補正ではなく、LLMによる人格上の戦術判断とする。追加のAI呼び出しは行わない。
- キャラクター設定には、真占い師と真霊媒師それぞれの結果あり／結果なしの公開意欲・注目耐性・時機・判断文、狂人と人狼それぞれの0人盤面の騙り意欲・2人目になる対抗意欲・3人目以降になる混雑許容・注目耐性・自己保全・圧力反応・好む騙り役職・時機・判断文、人狼だけの仲間側露出への警戒、一度選んだ作戦と公開主張を維持する方針を持つ。
- 0〜100の値はLLMへ人物傾向を伝える共通尺度で、0〜19はほぼ選ばない、20〜39は明確な非常時だけ、40〜59は利益と代償を具体的に比較、60〜79は条件が合えば積極的、80〜100は人物として強く好むと解釈する。実AIの確率抽選には使わず、MockAIだけが同じ設定から決定論的な反復試験を行う。
- v4は現在の占い師・霊媒師CO人数と、候補ごとに自分が何人目になるかを明示する。自分が2人目の場合だけ対抗意欲を使い、3人目以降は混雑許容を優先する。認可候補が存在すること自体をCO理由にせず、注目・混雑・仲間側露出の代償と人物固有の利益を比較させる。
- v4の`claimIntent`は、`information_duty`、`counterclaim`、`avoid_crowding`、`protect_team`など、人格と盤面のどの要素を決定打にしたかを非公開の`basis`へ一つ記録する。自由な思考過程は保存せず、公開イベントにも出さない。
- 真占い師は `人狼` という結果や既存対抗があれば直ちに名乗り、`人狼ではない` という結果だけでも1日目に名乗ることを標準とする。本人が待つ判断をしても公開議論全体のturn 6から必須の発言キューへ載せ、返答優先と2発言クールダウンを守って発言可能になった最初の機会に公開する。真霊媒師は公開処刑結果と対抗状況を見て判断し、遅くとも3日目には名乗る。
- 狂人には一般的な9人人狼の基本戦術として占い師騙りを有力な初期方針として与えるが、最終的な騙り役職・時機・潜伏は人物像と公開状況から本人が選ぶ。人狼2名もそれぞれ本人が判断するが、一方の人狼がすでに役職を名乗った後は、もう一方の新規役職主張をエンジンが禁止して無秩序な二重騙りを防ぐ。狂人は人狼位置を知らず、人狼は公開されていない狂人位置を知らない情報境界を維持する。
- `claimIntent` はAI呼び出しキャッシュに保存して復旧時に同じcall keyから再利用し、公開イベント、公開ビュー、ログへは出さない。待つ場合は再検討する公開上のtriggerを指定し、公開状況が変わった理由なしに気まぐれで方針を反転させない。公表した役職・対象・日・判定は従来どおりclaim ledgerで固定する。
- 偽結果は秘密情報漏洩と矛盾を防ぐため、エンジンが合法な候補として生成し、LLMは認可された役職と候補だけを選ぶ。狂人用候補の生成には本当の人狼位置を渡さず、人狼用候補も自分を対象にせず、日をまたいで同一対象の判定を変更しない。
- 保存済み `claims v1` / `claims v2` / `claims v3` の確率・seat補正・時機・人格判断契約は変更せず、各試合の `match_created.rules.claims` markerに従って同じ挙動を復旧する。v3の旧キャラクタースナップショットにv4項目がない場合は読み込み時に補完するが、v3の判断promptでは使わない。
- 一度公開した役職と結果は公開イベントからclaim ledgerとして導出し、日をまたいで同一対象・同一日の矛盾や存在しない結果を許さない。v1では撤回、訂正、スライドCO、狩人CO、村人の役職騙りを扱わない。
- 公開期限や真役職の人狼結果により必須となった役職主張は、動的キューの次話者候補へ直接入れる。任意の騙り判断のためにintent pollを追加せず、合意対象本人の追加反論を除く1人2回と1日2 pollの上限は維持する。
- AIへは「判断材料として知ってよい事実」と「ゲーム内で公に主張してよい内容」を分離して渡す。認可された偽主張は許可するが、認可外の対象・日・結果や、他者の本当の役職を知っているような判断は禁止する。
- 構造化claimを出さず役職と結果を伏せる場合は、能力結果だけでなく「村人だと確認できている」「結果はあるが今は言えない」のような確認済み正体・非公開結果の匂わせも禁止する。公開情報に基づく「村人らしく見える」などの推理は許可する。
- 必須主張がschema、本文、認可内容と一致しない場合は有界回数だけ同一モデルで再試行し、直らなければ試合を明示停止する。秘密情報、能力結果、役職主張に関わらない付随メタデータだけの不一致は、本文から裏付けられない `suspicion`、`voteIntent`、`questionTopic`、`boardAnalysis` を安全側へ破棄・格下げし、再試行で試合を止めない。必須主張を黙って削る、ランダム行動へ置換する、別モデルへフォールバックする処理は禁止する。

### 3.6 投票と同数

- 棄権不可。自分には投票できない。
- 投票は情報上同時に行う。先に投票したAIの票を後続AIへ見せない。
- 投票判断にも公開の証拠台帳を渡し、議論中の投票予定人数や多数派へ合わせること自体を根拠にせず、能力結果、発言、反応、相互関係を独立に比較させる。
- 投票時は同じ論点を繰り返した人数を独立した複数証拠として数えず、台帳上位の合法候補2人について、公開能力結果、役職主張と対抗、発言上の論点、本人の応答を並べてから選ばせる。役職を名乗った人物の言い間違い・説明の拙さと、その人物が公表した能力結果の内容は別々に評価させる。これは真役職を保護したり多数候補を降格したりする補正ではなく、比較手順だけを与える。
- 投票・決選投票の `statedReason` では、新しい役職主張や未公開能力結果を公表できない。昼の構造化claimを経ていない自己役職主張・自己能力結果を検出した場合、投票先は変えず、理由全文を固定の中立文へ置換して `reasonSanitized=true` を内部イベントへ記録する。理由本文からclaim ledgerを生成・更新せず、追加AI呼び出しもしない。
- 全票確定後、誰が誰に入れたかを一括公開する。
- 同数最多の場合は、その同数候補だけで決選投票を1回行う。
- 決選も同数なら、その日は処刑なしとする。
- 処刑なしも一般的な同票処理の一つであり、独自演出やランダム処刑は採用しない。

### 3.7 夜

処刑後、以下の順に処理する。

1. 生存する霊媒師へ、その日に処刑された人物が人狼か否かを非公開通知する。処刑なしなら `判定対象なし`。
2. 生存人狼が2名なら非公開会話を2周行う。生存人狼が1名だけの場合は1回だけ、会話modeを `monologue` とし、死亡した仲間への相談や返答要求ではなく、自分の読みと襲撃方針を口にする独り言として生成する。構造化応答でも `addressedTo=null`、`requestsReply=false` を強制する。
3. 各人狼が襲撃候補を選ぶ。
4. 生存占い師が占い対象を選ぶ。
5. 生存狩人が護衛対象を選ぶ。
6. 夜行動を同時解決する。
7. 襲撃先と護衛先が一致すれば犠牲者なし。それ以外は襲撃先が死亡する。
8. 次の昼へ進む。

### 3.8 人狼の襲撃

- 人狼自身と人狼仲間は襲撃対象にできない。
- 狂人は人狼から見ても正体不明なので、襲撃対象になり得る。
- 人狼2名の選択が一致すれば採用する。
- 不一致なら、追加の最終決定判断を座席番号の小さい生存人狼へ1回だけ依頼する。候補は両者が選んだ2名に限定する。
- 人狼1名なら単独で決める。
- `werewolf_chat` は `dialogue` / `monologue` をpayloadへ記録し、GM視点と終了後の答え合わせでは単独時を「人狼の独り言」と表示する。進行中の公開視点では生存人狼数を漏らさないよう、どちらも「人狼の夜会話」とだけ表示する。

### 3.9 占い師

- 自分は占えない。
- 死亡者は占えない。
- 同じ人物を再度占えるが、プロンプトでは未占い人物を優先するよう求める。
- 結果は `人狼` / `人狼ではない` の二値。
- 狂人は `人狼ではない`。

### 3.10 霊媒師

- 前日に処刑された人物だけを判定する。
- 結果は `人狼` / `人狼ではない` の二値。
- 狂人は `人狼ではない`。
- 襲撃で死亡した人物は判定しない。
- 結果は本人だけが知り、昼に共有するかはAIが判断する。

### 3.11 狩人

- 自分を護衛できる。
- 前夜と同じ対象を連続護衛できない。
- 連続護衛禁止によって合法対象がなくなる場合のみ解除する。
- 護衛成功時も、誰を護衛したか・狩人が誰かは公開しない。
- 狩人本人には、自分の過去の護衛先と、その護衛が成功したかを以後の判断コンテキストで非公開通知する。

### 3.12 死亡・役職公開

- 死亡者は以後、発言・投票・夜行動を行わない。
- 死亡者は能力・襲撃の対象にならない。
- 処刑・襲撃時に役職を公開しない。
- 試合終了時だけ、全員の役職と全非公開行動を公開する。
- 死者チャットはMVPに含めない。

### 3.13 停止性と異常系

- 第9日終了でも決着しなければ、異常フラグ付き引き分けとする。
- キャラクターごとに選択した言語モデルへの物理API呼び出し合計が240回へ到達する前に `aborted_budget` で停止する。
- AI判断失敗時はゲームを明示的に停止し、ランダム行動や別モデルで続行しない。

## 4. AI判断

### 4.1 言語モデルの選択

キャラクター編集画面で、各キャラクターが使うOpenAIまたはGeminiと、そのプロバイダーに対応する推論設定を選択する。9人それぞれのプロバイダー、サーバー設定のモデル名、推論設定を試合へスナップショット保存し、復旧時も発言者ごとに同じ組み合わせを使う。新規試合画面から試合全体を一括変更しない。

- OpenAIモデル: `gpt-5.6-luna`固定。`reasoning.effort`は`none / low / medium / high / xhigh / max`から選択し、既定は`low`。
- Gemini既定モデル: `gemini-2.5-pro`。サーバー環境変数`GEMINI_MODEL`でモデル名を変更できる。
- Gemini 2.5 Proの推論設定は`thinkingBudget`として保存し、自動`-1`または`128〜32768`の整数を許可する。画面は自動、128、1024、4096、8192、16384、32768をプリセット表示する。
- 1試合内でOpenAIとGeminiを併用できる。キャラクターが選択したプロバイダーが利用不能でも、別プロバイダーやランダム行動へフォールバックしない。

対象:

- 昼の発言。
- 役職を名乗るか、結果を共有するかどうか。
- 投票、決選投票。
- 人狼会話、襲撃候補、襲撃先の最終決定。
- 占い対象、護衛対象。
- 狂人の騙り・誘導を含む全戦略判断。

AI判断ではないもの:

- 配役、合法対象、昼夜遷移、票集計、勝敗、霊媒結果、占い結果、護衛結果。
- これらは決定論コードで処理する。

別プロバイダー・別モデルへのフォールバックは禁止。選択したモデルが利用不能なら停止する。

### 4.2 実AIの利用制限と費用管理

- 開発中の既定AIは必ず `MockAI` とする。
- Unit、Integration、Leak、E2E、UI確認、デバッグ、CI、反復シミュレーションではOpenAI／Gemini APIを一切呼ばない。
- 実AIクライアントの実装とテストには、固定レスポンスを返すMock transportを使う。
- `--ai real` を明示し、かつ環境変数 `ALLOW_REAL_AI=1` がある場合だけ実AIを許可する。片方でもなければ起動を拒否する。
- 実AIは、lint、typecheck、全自動テスト、MockAI 30試合、build、Docker buildがすべて成功した後の**最終受け入れ試験**でのみ使用する。
- 通常の最終受け入れ試験は実AIによる1試合完走だけとする。ただし昼議論の品質を変更した場合は、マスターの明示指示があるときに限り、1日目の処刑までを最大3試合、順番に実行して反復、灰評価、盤面進行、投票予定を検証できる。これは勝率・ゲームバランス測定へ流用しない。
- 実AI試験が失敗した場合は結果と消費呼び出し数を保存する。原因修正後はMockAIで再検証を完了してから、同じ受け入れ範囲を再実行する。
- READMEと`.env.example`にも、実AIは有料であり最終確認専用であることを明記する。

### 4.3 OpenAI／Gemini SDK

SDKの自動リトライは無効化し、アプリ側だけで回数を管理する。

```ts
const client = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY,
  maxRetries: 0,
  timeout: 60_000,
});

const response = await client.responses.parse({
  model: 'gpt-5.6-luna',
  reasoning: { effort: savedOpenAiReasoningEffort },
  input: [
    { role: 'system', content: systemPrompt },
    { role: 'user', content: decisionPrompt },
  ],
  text: { format: zodTextFormat(schema, 'werewolf_decision') },
});
```

Geminiは`@google/genai`をサーバー側だけで使い、Zodから変換したJSON SchemaによるStructured Outputsを要求する。`gemini-2.5-pro`ではキャラクターの`llm.thinkingBudget`を`thinkingConfig.thinkingBudget`へ渡す。

### 4.4 情報分離

全AIが受け取る情報:

- 一般ルール。
- 自分の座席と役職。
- 全員の座席・生死。
- 公開済み発言、GM発表、記名投票。
- 現在の日・フェーズ・求められている判断。

役職固有:

- 人狼: 仲間、生存人狼だけの夜会話、過去の最終襲撃先と襲撃の成否。
- 占い師: 自分の全占い履歴と結果。
- 霊媒師: 自分の全霊媒結果。
- 狩人: 自分の全護衛履歴と護衛の成否。
- 狂人: 追加情報なし。人狼を推理して支援する。
- 占い師・霊媒師・狂人・人狼: 自分にだけ与えられた役職主張directive。directiveは公開主張として認可された役職・結果だけを含み、真偽を表す語や他者の秘密役職を含めない。

渡してはいけない情報:

- 他人の本当の役職。
- 他役職の非公開結果。
- 未公開票。
- 人狼以外への人狼会話。
- GM視点用の判断メモ。
- 言語モデルの内部reasoning。

### 4.5 エージェントの同一性と編集

- 下記の9人を初期プリセットとし、キャラクター編集画面で名前、肩書き、一人称、人格、感情、話し方、判断の癖、禁止スタイル、6役職の行動方針、既定の呼称スタイル、任意の個別呼称、立ち絵、LLM、推論設定、TTSプロバイダー、VOICEVOX／AivisSpeech話者を編集できる。
- 各プリセットはJSONで書き出し・読み込みでき、初期設定へ個別に戻せる。JSON読み込みと立ち絵アップロードは、クリック選択とドラッグ＆ドロップの両方に対応する。
- JSONの作成方法、全フィールド、画像とTTS話者の指定方法は [`docs/character-preset-json.md`](character-preset-json.md) に記載する。
- JSON読み込み時はCLIの `npm run character:validate` と同じ厳密スキーマで全項目を検証し、不正なプリセットを編集内容へ反映しない。問題数、フィールドパス、理由をまとめて表示する。
- 共有用JSONの `seat` と `tts.voice.seat` は空文字、`addressBook` は空オブジェクトとする。参加者未定でも`defaultAddressStyle`で表示名／名字／下の名前と敬称の大まかな既定ルールを指定する。読み込み先はキャラクター編集画面で現在選択中の保存枠に固定し、内容確認後の保存でその枠だけを上書きする。試合開始時には全キャラクターをseedで決定論的に席へ配置し直す。個別呼称は相手キャラクターへ追従し、`addressBook`の指定を既定ルールより優先する。
- キャラクターの実行設定は排他的な判別可能unionとする。LLMは`llm: { provider: 'openai', reasoningEffort } | { provider: 'gemini', thinkingBudget }`、TTSは`tts: { provider: 'voicevox' | 'aivisspeech', voice }`のいずれか1構成だけを保存し、選択していないプロバイダー用設定はJSONへ持たせない。旧DBと旧試合スナップショットのフラット形式は読み込み時に正規化する。
- 既存の人格を持つAIキャラクター本人へ作成を依頼する手順は [`docs/ai-character-preset-authoring.md`](ai-character-preset-authoring.md) に記載する。席は質問せず空欄とし、画像、LLM、TTS、他キャラクターへの大まかな呼び方を対話相手へ必ず確認する。LLM・推論設定・TTSプロバイダー・既定の呼称スタイルはキャラクターJSONへ含め、検証結果とともに返す。
- プレイヤー名は `名取 澪`、`天満 ひなた`、`宮下 さくら`、`雨宮 しずく`、`神崎 レナ`、`黒田 剛`、`真壁 陽太`、`福本 源蔵`、`久遠 ひより`。
- 各人に行動原理、内面の矛盾、欠点、対人バイアス、感情傾向、発言量の差を与える。
- 共有用プリセットの個別呼称は空でよいが、`defaultAddressStyle`で表示名、名字、下の名前、さん／ちゃん／くん付け、呼び捨ての大まかな既定ルールを指定する。9人の組み合わせ確定後は、人格と距離感に応じて相手ごとの呼称を上書きできる。試合開始時の席配置に呼称を追従させ、Agent番号呼びは禁止する。
- 既定キャラクターの保存枠を別人格へ置き換えた場合、他の初期キャラクターが旧相手へ持っていた既定の個別呼称を新規試合のスナップショットから除去する。置き換え後に明示設定した新しい個別呼称は保持し、それ以外は各話者の`defaultAddressStyle`へフォールバックする。
- 人格は推理能力や秘密情報を増やさない。同じ公開情報でも、好き嫌い、見栄、迷い、思い込みによって判断と表現が変わる。
- 口調だけを表面的に変えるのではなく、一人称、文長、敬体／常体、口癖、感情の出方、疑いの持ち方、撤回のしやすさを人物ごとに分離する。正確な盤面整理は全員が同じ口調で説明することを要求しない。
- 欠点は装飾ではなく判断の癖として与える。人格の偏りをその場で打ち消す注意書き（「〜しすぎないように」「固執せず」など）を人格定義と役職行動方針に書かない。偏りが招く損はゲームの一部として許容し、公開情報の事実関係、能力結果と役職主張の正確さ、秘密情報の境界だけを全員共通の床とする。回復の仕方（引かない、翌日だけ変える、無反省で切り替える、潔く訂正する）も人物ごとに分離する。
- 長い進行・公開状態・漏洩防止指示によって人格が薄まらないよう、発言生成のシステムプロンプト末尾で、対象人物の声とリズム、判断の癖、避ける話し方を最終演技契約として再提示する。この契約も存在しない発言・反応・投票・能力情報の創作を許可しない。
- 9人それぞれに6役職分、合計54通りの役職行動方針を持たせる。現在の座席と実役職をキーに対応する方針を1件だけ機械的にシステムプロンプトへ挿入し、発言、投票、役職を明かす時機、占い、護衛、襲撃の選び方へ人格を反映する。
- 役職行動方針は一般ルール上の能力や情報を増やさない。とくに狂人へ人狼の正体を与えず、狩人の連続護衛禁止を上書きしない。
- 全体要約や模範解答調の発言を避け、直前の相手へ自然に反応する。ただし口癖や欠点を毎回わざとらしく演じない。
- 個別人格は一般的な人間関係の範囲に留め、独自職業、固有能力、物語上の世界設定を持ち込まない。

### 4.6 Structured Outputs

```ts
const SpeechDecision = z.object({
  speech: z.string(),
  addressedTo: z.enum(legalSeatIds).nullable(),
  requestsReply: z.boolean(),
});

const SpeechIntentDecision = z.object({
  urgency: z.union([z.literal(0), z.literal(1), z.literal(2), z.literal(3)]),
  motivation: z.enum(['reply', 'question', 'challenge', 'new_information', 'clarify', 'none']),
  targetSeat: z.enum(legalSeatIds).nullable(),
});

const TargetDecision = z.object({
  targetSeat: z.enum(legalSeatIds),
  statedReason: z.string().max(120),
});
```

- 対象は名前ではなく `seat-1`〜`seat-9`。
- 合法対象のみを動的enumへ入れる。
- `statedReason` は短い説明であり、chain-of-thoughtを要求しない。第0夜の真占い対象はこのschemaを使わずengineが決定する。1日目夜以降の占い、投票、襲撃、護衛では従来どおり使う。
- 発言、投票、襲撃、占い、護衛をstrict schemaで処理する。

### 4.7 リトライ

- 再試行可能な失敗は種類をまたいで合算し、1つの判断につきLLM呼び出しを最大3回とする。1回目と2回目の失敗後に1秒、2秒のバックオフを使い、3回目の失敗で試合を明示停止する。
- 接続失敗、タイムアウト、408、409、429、5xx、refusal、structured output、parse・Zod検証失敗は再試行可能とする。再試行不能な400系は即時停止する。
- ローカルの発言・主張契約違反は具体的な修正指示を次の呼び出しへ付ける。補正可能な付随メタデータ不一致はAPIを再試行しない。
- 400、401、403、404は即 `paused_error`。
- 再試行可能な失敗が3回に達した場合も `paused_error`。
- 再開は同じ `gpt-5.6-luna` で同じ判断を再試行するだけ。

### 4.8 API呼び出し上限

- `MAX_API_CALLS = 240`。
- リトライを含む物理HTTP呼び出し数を数える。
- 呼び出し直前にDBへカウンタと `in_flight` を保存する。
- 上限へ到達する呼び出しは送信しない。

### 4.9 クラッシュ時の境界

- 保存済みAI応答は再起動後に再利用する。
- 言語モデル応答受信後・DB保存前にプロセスが落ちた可能性がある `in_flight` は自動再実行しない。
- `ambiguous_ai_call` として停止し、人間が再試行を選ぶ。
- この狭い障害窓では二重課金の可能性はあるが、ゲームイベントは一度だけ確定する。

## 5. 観戦UI

### 5.1 視点

| 情報 | 公開視点 | GM視点 | 終了後 |
|---|---:|---:|---:|
| 発言、投票、死亡 | ○ | ○ | ○ |
| 非公開イベントの実施（内容は伏せる） | ○ | ○ | ○ |
| 本当の役職 | × | ○ | ○ |
| 人狼会話 | × | ○ | ○ |
| 襲撃・占い・霊媒・護衛 | × | ○ | ○ |
| statedReason | × | ○ | ○ |

- 公開データはサーバー側でホワイトリスト射影する。
- 進行中の公開視点では、非公開イベントを固定ラベルだけの `private_action` に射影し、「人狼確認が行われました。」のように実施事実だけを表示する。元のpayload、対象、判定結果、発言本文、理由、audienceは含めない。
- 進行中の公開視点では `statedReason` を含む非公開判断理由をpayloadから除去する。
- 秘密情報をブラウザへ送ってCSSで隠す方式は禁止。
- ライブ中に公開／GM視点を切り替えられる。
- `finished`、`aborted`、`aborted_budget` の終了後は、公開視点でも全役職・人狼会話・夜行動・判断理由を答え合わせとして開示する。

### 5.2 画面

| パス | 内容 |
|---|---|
| `/` | 新規試合開始、seed・速度、試合一覧 |
| `/match/[id]` | ライブ観戦 |
| `/replay/[id]` | 保存イベントのリプレイ |

ホーム:

- `AI人狼` テキストロゴ。
- seed入力。空なら自動生成。
- 速度: ゆっくり3秒、標準1.5秒、最速0秒。
- `AI人狼を開始` ボタン。
- `キャラクターを編集` 導線。9保存枠の人格、立ち絵、LLM、推論設定、TTS、席未定JSONの入出力、個別初期化を行える。
- 試合一覧: 日時、seed、状態、勝者、日数。

ライブ:

- ヘッダー: 日数、昼夜・フェーズ、生存数、公開／GM視点、状態。GM視点と終了後は生存人狼数も表示する。
- 中央: 固有名を持つ9人のカード。座席、生死、死亡日と死因、最新発言、直近の投票先、GM視点と終了後の役職バッジ。発言中のキャラクター画像、名前、セリフをカード一覧の上に大きく表示し、pause中も同じ発言を保持する。進行中は発言ステージと3×3の全カードを下部操作バーより上へ常時収め、発言ステージと投票結果パネルは同時に表示しない。
- 右: 公開された役職主張をコンパクトにまとめ、その下に日区切りと重要イベントの視覚的階層を持つ独立スクロールの時系列ログを置く。発言、記名投票、処刑、夜明け、GM秘密、終了後に開示された秘密を区別する。
- 下: pause/resume、速度、中断。pauseは新規AI判断に加え、ブラウザの再生中音声、発言キュー、表示位置を停止し、resumeで同じ音声位置から再開する。pause前から実行中のAI応答は完了を待つが、resumeまでイベント化しない。演出開始または手動pauseが音声の再生開始と競合してブラウザが再生を拒否した場合、その発言を完了扱いにせず、resume後に同じ音声を再試行する。発言音声は再生順を保ったまま次の2発言まで先行合成するが、TTS Engineの過負荷とタイムアウトを避けるため、クライアント内とサーバー上の同一Engine単位の両方で合成リクエストを1件ずつ直列実行する。発言終了後は現在の話者と表示位置を1秒保持してから次の発言またはアクションへ進む。VOICE無効化または観戦画面の破棄時は未再生の先行合成をキャンセルする。
- 投票は投票者名、投票先、正規化した得票バーを表示し、GM視点と終了後は判断理由も展開できる。
- 公開視点で非公開の夜処理中や同時投票中も、直前の公開イベントから表示フェーズを安全に導出し、停止と誤認させない。
- `paused_error`ではモデル名、失敗フェーズ、安全に保存したreasonを日本語化した具体的な原因、再試行を表示する。
- 観戦画面内のルールガイドで、9人の配役、勝利条件、昼夜進行、公開／GM視点の違いを確認できる。
- 新規試合は第0夜、1日目の順に開始を画面全面で示す。1日目の演出は第0夜の占い結果確認後、最初の発言より前に表示し、全画面演出中はVOICEと発言表示を待機させる。以後も夜明け、開票、処刑を2.4〜3.6秒の切替演出で示す。夜明けは襲撃犠牲者名または犠牲者なしを大きく表示し、一拍置いてから新しい日数を表示する。通常の連続する演出間には0.6秒、開票から次の演出までには投票結果を読むため5秒の間を設ける。投票開始は全画面演出にせず、画面上のフェーズ表示だけで示す。当日の開票前に前日の投票結果を注目パネルや人物カードへ再表示しない。襲撃犠牲者と処刑者は名前を表示し、それぞれ専用のCC0効果音を鳴らす。効果音は独立してON/OFFと音量調整ができ、設定は再読込後も維持する。再接続・視点切替で過去イベントを重複再生しない。
- 終了時は勝利陣営に加え、陣営別の全配役、各人の生存・処刑・襲撃、個人勝敗をミニエピローグとして表示する。狂人は判定上の人間かつ人狼陣営であることを併記する。

リプレイ:

- ライブと同じコンポーネントを使う。
- イベントseqのシークバー。
- 前後イベント、前後フェーズ、再生、停止、速度。

イベント封筒:

```ts

  matchId: string;
  seq: number;
  day: number;
  phase: Phase;
  type: T;
  visibility: 'public' | 'private';
  audienceSeats: number[];
  payload: P;
  createdAt: string;
}
```

新規試合の `match_created` は `rules.claims = 'v4'`、`rules.discussion = 'v3'`、`rules.nightZero = 'uniform'` を持つ。`nightZero` markerがある試合だけengine所有の第0夜選択、固定無情報事実、投票理由サニタイズを使う。markerのない保存済み試合は、AIの `d0-seer` 応答を含む従来経路とpayloadを再生して復旧時のevent divergenceを防ぐ。`discussion_speech` は従来の台詞、宛先、返答要求、`claim` に加えて、v3では公開発言の自己分類である `structure` を保存し、変更のない投票予定再宣言を主要貢献から格下げした場合だけ `contributionDemoted=true` を保存する。`claim` は `null` または `claimedRole` と結果配列だけを含む。`claimIntent`と`basis`はAI呼び出しキャッシュだけに保存し、公開payloadへ含めない。`stage = 'opening'` はその人物の当日1回目、`stage = 'free'` は2回目以降の発言を表し、全体の固定一巡は表さない。3回目は合意対象本人にだけ許す追加反論である。`rules.discussion = 'v2'` の保存済み試合はv2の発言schema、プロンプト、キュー、payloadを変更せず再開する。`rules.claims = 'v1'` / `'v2'` / `'v3'` の保存済み試合は各markerの方針で再開する。`rules.discussion` markerのない旧試合は従来の固定一巡経路で再開する。公開claim ledger、観戦UI、リプレイは保存済みの `discussion_speech.claim` だけから導出し、真役職・真結果・非公開方針を参照しない。v3の `structure` と `contributionDemoted` は当面、公開観戦画面へ表示せず、議論制御と受け入れ評価にだけ使う。

### 7.7 SQLite

- WAL。
- foreign keys ON。
- 1判断のAI記録、生成イベント群、match更新を1トランザクションで保存する。

### 7.8 決定論

- `Math.random()`禁止。
- `rand(seed, purposeKey)` で用途ごとに乱数を導出する。
- 配役シャッフル、昼の最初の話者と指名のない次話者、GM固定文選択に使う。
- AI出力自体は決定論とみなさない。
- 同seed＋MockAIならイベントpayload列を一致させる。
- 新規試合の第0夜占い対象は `stableIndex(seed, 'd0-seer', legalTargets.length)` で決め、AI出力に依存させない。真占い師とseed由来の偽占い結果で対象分布にAI由来の差を作らない。
- 保存済みv1の役職主張方針と偽結果は `claims/v1/` 名前空間のpurpose keyから純粋導出し、確率表と導出ロジックを変更しない。新規v2では占い師系の登場スロットだけ `claims/v2/slot/seer/` から導出し、その他の騙り担当・役職・偽結果の一貫性は凍結済みロジックを再利用する。

### 8.6 最終実AI受け入れ試験

以下の試験は、8.1〜8.5、build、Docker buildがすべて成功した後、全工程の最後に1回だけ実行する。

```bash
ALLOW_REAL_AI=1 npm run sim -- --matches 1 --ai real --seed-base 1000
```

合格条件:

- 1試合がクラッシュせず終端する。
- 使用モデルが `gpt-5.6-luna` である。
- 全リクエストが `reasoning.effort=low` である。
- 240回のAPI上限を超えない。
