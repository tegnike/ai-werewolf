## 0. 最重要方針

これは**一般的な人狼ゲームをAI同士にプレイさせるアプリ**である。独自ゲームへ改変しない。

- ゲーム名は画面上もREADME上も、単に `AI人狼` / `AI Werewolf` とする。
- 役職名は一般的な `村人・人狼・占い師・霊媒師・狩人・狂人` をそのまま使う。
- 人狼、村人、占い師などを別の固有名詞へ言い換えない。
- 独自の世界観、物語、職業設定、キャラクター設定を付けない。
- AIは9人の固定名と個別人格を持つプレイヤーとして参加する。ただし独自役職・物語・世界観は付けない。
- 一般的な9人人狼のルールを採用し、ルール文、コード、画像、UIは独自制作する。

本書は、Fableが提案した観戦UXと堅牢な技術構成を、マスターの訂正に従って**標準人狼へ全面的に戻した修正版**である。以前の独自タイトル・独自役職・独自世界観はすべて破棄し、実装へ持ち込まない。

## 1. プロダクト

### 1.1 名称と目的

- 日本語名: `AI人狼`
- 英語名: `AI Werewolf`
- リポジトリ: `ai-werewolf`

> 9体のAIエージェントが、一般的な人狼ゲームを最初から最後まで自律プレイし、人間が公開視点またはGM視点で観戦できるWebアプリ。

### 1.2 MVP

- AIプレイヤー9名固定。
- 9名は固定の名前、行動原理、欠点、感情傾向、話し方を持つ。
- アプリがGMを担当する。
- 人間は開始、観戦、一時停止、再開、中断、リプレイのみを行う。
- 昼の議論、投票、夜の人狼会話・襲撃・役職行動を完全自動進行する。
- 公開視点とGM視点を切り替えられる。
- SQLiteへ全イベントを保存し、試合後にリプレイできる。
- ローカル／セルフホスト前提。Docker対応。
- 観戦UIはPCのデスクトップブラウザ専用とする。
- 画像は実装セッションがCodex組み込み`imagegen`で生成する。
- ON/OFF・個別音量調整可能なオリジナルBGMと、9名別々のVOICEVOX読み上げを備える。
- VOICEVOXの話速は9名共通で `speedScale=1.1` とする。
- 人名の漢字表記はVOICEVOXユーザー辞書APIへ固有名詞として同期し、話者ごとに名字・名前・敬称が変わっても読みを統一する。同期は冪等とし、アプリ外の辞書項目を変更しない。

### 1.3 非ゴール

- 人間プレイヤー参加。
- 役職・人数のカスタマイズ。
- アカウント、課金、ランキング、マッチメイキング。
- 多言語対応。
- スマートフォン向けデザイン、タッチ操作最適化、モバイル用レイアウト。
- サーバーレス、複数サーバー、水平スケール。
- 特定の既存人狼製品との互換性・再現。

## 3. 標準9人ルール

### 3.1 配役

| 役職 | 陣営 | 人数 | 能力 |
|---|---|---:|---|
| 村人 | 村人陣営 | 3 | 特殊能力なし。会話と投票で人狼を探す |
| 人狼 | 人狼陣営 | 2 | 仲間を知り、夜に相談して1名を襲撃する |
| 占い師 | 村人陣営 | 1 | 毎夜1名が人狼か否かを知る |
| 霊媒師 | 村人陣営 | 1 | 前日に処刑された人物が人狼か否かを知る |
| 狩人 | 村人陣営 | 1 | 毎夜1名を人狼の襲撃から守る |
| 狂人 | 人狼陣営 | 1 | 人間だが、人狼陣営の勝利を目指す。特殊能力なし |

重要:

- 人狼2名は互いを知る。
- 人狼は狂人が誰か知らない。
- 狂人は人狼が誰か知らない。
- 占い師が狂人を占うと `人狼ではない` と出る。
- 霊媒師が処刑された狂人を判定しても `人狼ではない` と出る。
- 狂人は人数判定上は非人狼として数えるが、人狼陣営が勝てば勝利する。
- 狂人自身が死亡していても、人狼陣営が勝てば狂人の勝利とする。

### 3.2 勝利条件

- 村人陣営勝利: 生存する人狼が0名になる。
- 人狼陣営勝利: 生存人狼数が、生存する非人狼人数以上になる。
- 狂人は勝敗判定の「人狼数」には含めない。
- 判定は夜明け後と処刑後に行う。

例:

- 人狼1＋村人系1: 人狼勝利。
- 人狼1＋狂人1: 人狼1、非人狼1なので人狼勝利。
- 人狼1＋狂人1＋村人1: 人狼1、非人狼2なので継続。
- 人狼0＋狂人1: 村人勝利。

### 3.3 第0夜

初日の議論材料を残すため、第0夜は襲撃なしとする。

1. GMがゲーム開始を宣言する。
2. 人狼2名に仲間を非公開通知する。
3. 人狼2名が1回ずつ非公開で顔合わせ会話をする。襲撃先は選ばない。
4. 占い師が自分以外の1名を選んで占う。
5. 占い結果を占い師本人だけへ通知する。
6. 霊媒師と狩人は行動しない。
7. 第1日へ進む。

### 3.4 昼

1. 2日目以降は、前夜の犠牲者名、または犠牲者なしをGMが公開する。襲撃のない第0夜明けは省略し、1日目は最初の発言から始める。
2. 勝敗判定。
3. 固定順を設けず、動的な話者選択による昼の議論を行う。生存者全員に最低1回の発言機会を保証する。
4. 生存者全員が同時投票する。
5. 記名投票結果を公開する。
6. 最多得票者を処刑する。
7. 処刑者の名前だけ公開し、役職は試合終了まで公開しない。
8. 勝敗判定。
9. 未決着なら夜へ進む。

### 3.5 発言

- 昼の議論は固定の開始発言一巡を持たず、最初から最後まで1本の動的な発言キューで進行する。最初の話者と、指名・必須主張・発言希望がない場合の次話者はseedから決定論的に選ぶ。
- 発言者が特定の生存者へ返答を求めた場合は、構造化結果の `addressedTo` と `requestsReply` を正本とし、発言上限へ達していないその相手を次の話者にする。台詞本文の文字列解析で指名を判定しない。
- 次話者の優先順位は、返答を求められた人、必須の役職主張・対抗がある人、未発言者、発言希望者、seedによる選択の順とする。
- その日の先頭発言者は、まだ起きていない当日の沈黙、反応、便乗、返答遅延を観察したように話さない。公開済みの能力結果、自分がその発言で公開する能力結果、前日までの発言などの根拠がない場合は、疑い先・投票先・暫定評価を無理に作らず、確認したい論点、質問、今後の判断基準から自然に議論を始める。全話者は仮定・方針・弱い違和感を、具体的な告発へ勝手に強めない。
- 生存者全員が最低1回話し、通常は1人1日最大2回とする。3人以上の投票予定が集まった対象本人だけは、合意形成後の反論として1回追加できる。9人生存時は通常発言14回未満で終了せず、通常枠18回に追加反論枠1回を加えた最大19回で必ず終了する。
- 8人以下では最低発言数を `max(生存者数, ceil(14 × 生存者数 / 9))`、最大発言数を `min(18, 生存者数 × 2)` とし、人数に比例して議論量を縮小する。
- 発言結果は本文に加えて、実際に話を向けた相手と返答要求の有無を返す。最後の発言枠では新しい返答要求を禁止する。
- 最低発言数と全員1回を満たした後に発言の連鎖が止まったとき、1日最大2回、各回最大4人へ非公開の発言希望確認を行う。発言希望は強さ、動機、対象だけをstrict schemaで返し、台詞や長い思考過程を含めない。
- 発言希望確認は、台詞生成用の話し方・呼称・台詞例を省いた短いプロンプトと、直近30件の公開履歴だけを使い、入力トークンを抑える。
- 発言希望の候補は、まだ確認されていない人、発言回数が少ない人を優先し、同点はseedから決定論的に選ぶ。
- 最低発言数と全員1回を満たしたうえで発言希望がない、2回の確認を使い切る、または最大発言数へ達すると投票へ進む。
- 1発言は日本語200 Unicode code point以内。
- 空文字は `……（沈黙）` として扱う。
- 役職を名乗ること、占い結果・霊媒結果の共有、騙り、嘘、弁明、質問は自由。
- システムは役職の名乗りや結果報告の真偽を証明しない。
- 新規試合の `discussion v3` では、各発言が本文と一致する `primaryAct`、質問または回答の話題、暫定的な疑い先・根拠分類・根拠日 `evidenceDay`、投票予定を構造化して返す。勘だけなら `evidenceDay=null` とし、公開情報を根拠にする場合はその情報の日を指定する。今日まだ未発言の人物でも前日以前の公開情報は根拠にできるが、当日の未観測の態度は根拠にできない。これは発言者の公開した立場を記録するもので、真の役職や内部推論ではない。
- エンジンはv3の構造化発言だけから、既出質問と回答数、疑いの分布、投票予定、盤面整理などの貢献数を日ごとに畳み込む。本文の文字列解析や別のAI呼び出しは使わない。同じ質問分類が2回明示された後はその分類を閉じ、後続の構造化出力では新たな返答要求に使えないようにする。回答として話題へ触れることは妨げない。
- 後続話者には、候補ごとの投票予定人数だけでなく、疑いを出した人数、根拠分類、公開された占い・霊媒結果と同役職主張人数をまとめた証拠台帳と、不足している貢献候補を渡す。発言を根拠にした暫定評価の候補には、当日すでに発言した人物だけを含める。投票予定の人数自体は証拠として扱わせず、同じ質問の反復より、独自の灰評価、弁明、役職候補の扱い、処刑方針を優先させる。候補は台詞や結論を固定する台本にはしない。
- 同じ相手へ3人以上の投票予定が集まった後は、後続話者が同じ相手への投票予定を本文で追加宣言したり `voteIntent` へ設定したりしない。これは最終の非公開投票先を拘束せず、合理的な票合わせは妨げない。後続発言は増えた公開情報、被疑者への質問、反証、または未検討人物の検討へ使う。3人未満の段階で同一話者が投票先を変えず再宣言した場合は、本文と投票予定を保存するが `primaryAct` を `agreement` へ格下げし、主要な新規貢献として数えない。格下げは再試行を発生させず、`contributionDemoted=true` として記録する。
- 一般的な進行知識として、別々の対象への `人狼ではない` という結果は互いに矛盾しないこと、`人狼ではない` という結果だけの占い師候補2人を即座に処刑対象へ限定しないこと、`人狼` という結果が出た場合は対象本人と占い師候補を比較すること、霊媒師候補2人では順番に処刑する進行を検討することをv3へ与える。
- キャラクターの台詞では、能力結果や人物評価を「白」「黒」「白結果」「黒結果」「初手黒」などと省略しない。`人狼` / `人狼ではない` という構造化値は維持し、本文では「人狼だという結果」「人狼ではないという結果」のような自然な日本語で伝える。保存済みの公開発言に略語が含まれていても後続話者は模倣しない。
- 2人以上から疑いを向けられたか、3人以上から投票予定を向けられ、当日まだ1回しか話していない人物には、必須の役職主張より後、未発言者のseed選択より前に1回だけ弁明機会を与える。まだ一度も話していない集中先は、通常の未発言者より先に発言させる。明示的な返答要求による次話者指定はこれより優先する。
- 占い師本人には、各夜の占い対象と結果に加えて、その対象を選んだ時点の理由を非公開履歴として渡す。理由は公開イベントや他の参加者へ射影しない。

#### 3.5.1 役職主張・対抗CO・騙り

- 公開発言は、台詞とは別に任意の構造化主張を持てる。主張できる役職は占い師・霊媒師で、対象、対象夜／処刑日、本人が主張する `人狼` / `人狼ではない` の結果を含む。
- 構造化主張は真役職や真結果ではなく、その発言者が公に述べた内容である。真役職から自動生成せず、真占い師・真霊媒師・狂人・人狼を同じ発言契約と検証経路へ通す。
- 真占い師は `人狼` という結果、既存対抗、日数、seedで選ばれたCO時機によって公開を早め、遅くとも2日目には名乗る。真霊媒師は公開処刑結果と対抗状況を見て判断し、遅くとも3日目には名乗る。
- 狂人はseedと人格差から高めだが固定ではない確率で占い師または霊媒師を騙る。人狼2名のうちseedで選ばれた1名は状況依存で騙り、非担当の人狼も自分へ `人狼` という結果を出された場合などに緊急対抗し得る。全員潜伏、狂人対抗、人狼対抗をいずれも再現可能にする。
- 騙り方針、CO時機、偽結果はmatch seedから決定論的に導出し、公開イベントや別DBテーブルへ秘密状態として保存しない。狂人の偽結果生成には本当の人狼位置を渡さない。
- 一度公開した役職と結果は公開イベントからclaim ledgerとして導出し、日をまたいで同一対象・同一日の矛盾や存在しない結果を許さない。v1では撤回、訂正、スライドCO、狩人CO、村人の役職騙りを扱わない。
- 既存対抗への発言が必要な場合は動的キューの次話者候補へ直接入れる。intent pollを追加せず、合意対象本人の追加反論を除く1人2回と1日2 pollの上限は維持する。
- AIへは「判断材料として知ってよい事実」と「ゲーム内で公に主張してよい内容」を分離して渡す。認可された偽主張は許可するが、認可外の対象・日・結果や、他者の本当の役職を知っているような判断は禁止する。
- 構造化claimを出さず役職と結果を伏せる場合は、能力結果だけでなく「村人だと確認できている」「結果はあるが今は言えない」のような確認済み正体・非公開結果の匂わせも禁止する。公開情報に基づく「村人らしく見える」などの推理は許可する。
- 必須主張がschema、本文、認可内容と一致しない場合は有界回数だけ同一モデルで再試行し、直らなければ試合を明示停止する。秘密情報、能力結果、役職主張に関わらない付随メタデータだけの不一致は、本文から裏付けられない `suspicion`、`voteIntent`、`questionTopic`、`boardAnalysis` を安全側へ破棄・格下げし、再試行で試合を止めない。必須主張を黙って削る、ランダム行動へ置換する、別モデルへフォールバックする処理は禁止する。

### 3.6 投票と同数

- 棄権不可。自分には投票できない。
- 投票は情報上同時に行う。先に投票したAIの票を後続AIへ見せない。
- 投票判断にも公開の証拠台帳を渡し、議論中の投票予定人数や多数派へ合わせること自体を根拠にせず、能力結果、発言、反応、相互関係を独立に比較させる。
- 全票確定後、誰が誰に入れたかを一括公開する。
- 同数最多の場合は、その同数候補だけで決選投票を1回行う。
- 決選も同数なら、その日は処刑なしとする。
- 処刑なしも一般的な同票処理の一つであり、独自演出やランダム処刑は採用しない。

### 3.7 夜

処刑後、以下の順に処理する。

1. 生存する霊媒師へ、その日に処刑された人物が人狼か否かを非公開通知する。処刑なしなら `判定対象なし`。
2. 生存人狼が2名なら非公開会話を2周行う。生存人狼が1名だけの場合は1回だけ、会話modeを `monologue` とし、死亡した仲間への相談や返答要求ではなく、自分の読みと襲撃方針を口にする独り言として生成する。構造化応答でも `addressedTo=null`、`requestsReply=false` を強制する。
3. 各人狼が襲撃候補を選ぶ。
4. 生存占い師が占い対象を選ぶ。
5. 生存狩人が護衛対象を選ぶ。
6. 夜行動を同時解決する。
7. 襲撃先と護衛先が一致すれば犠牲者なし。それ以外は襲撃先が死亡する。
8. 次の昼へ進む。

### 3.8 人狼の襲撃

- 人狼自身と人狼仲間は襲撃対象にできない。
- 狂人は人狼から見ても正体不明なので、襲撃対象になり得る。
- 人狼2名の選択が一致すれば採用する。
- 不一致なら、追加の最終決定判断を座席番号の小さい生存人狼へ1回だけ依頼する。候補は両者が選んだ2名に限定する。
- 人狼1名なら単独で決める。
- `werewolf_chat` は `dialogue` / `monologue` をpayloadへ記録し、GM視点と終了後の答え合わせでは単独時を「人狼の独り言」と表示する。進行中の公開視点では生存人狼数を漏らさないよう、どちらも「人狼の夜会話」とだけ表示する。

### 3.9 占い師

- 自分は占えない。
- 死亡者は占えない。
- 同じ人物を再度占えるが、プロンプトでは未占い人物を優先するよう求める。
- 結果は `人狼` / `人狼ではない` の二値。
- 狂人は `人狼ではない`。

### 3.10 霊媒師

- 前日に処刑された人物だけを判定する。
- 結果は `人狼` / `人狼ではない` の二値。
- 狂人は `人狼ではない`。
- 襲撃で死亡した人物は判定しない。
- 結果は本人だけが知り、昼に共有するかはAIが判断する。

### 3.11 狩人

- 自分を護衛できる。
- 前夜と同じ対象を連続護衛できない。
- 連続護衛禁止によって合法対象がなくなる場合のみ解除する。
- 護衛成功時も、誰を護衛したか・狩人が誰かは公開しない。
- 狩人本人には、自分の過去の護衛先と、その護衛が成功したかを以後の判断コンテキストで非公開通知する。

### 3.12 死亡・役職公開

- 死亡者は以後、発言・投票・夜行動を行わない。
- 死亡者は能力・襲撃の対象にならない。
- 処刑・襲撃時に役職を公開しない。
- 試合終了時だけ、全員の役職と全非公開行動を公開する。
- 死者チャットはMVPに含めない。

### 3.13 停止性と異常系

- 第9日終了でも決着しなければ、異常フラグ付き引き分けとする。
- 物理OpenAI API呼び出しが240回へ到達する前に `aborted_budget` で停止する。
- AI判断失敗時はゲームを明示的に停止し、ランダム行動や別モデルで続行しない。

## 4. AI判断

### 4.1 モデル固定

すべてのAI判断はOpenAI Responses APIで以下へ固定する。

```ts
model: 'gpt-5.6-luna'
reasoning: { effort: 'low' }
```

対象:

- 昼の発言。
- 役職を名乗るか、結果を共有するかどうか。
- 投票、決選投票。
- 人狼会話、襲撃候補、襲撃先の最終決定。
- 占い対象、護衛対象。
- 狂人の騙り・誘導を含む全戦略判断。

AI判断ではないもの:

- 配役、合法対象、昼夜遷移、票集計、勝敗、霊媒結果、占い結果、護衛結果。
- これらは決定論コードで処理する。

別モデルへのフォールバックは禁止。モデルが利用不能なら停止する。

### 4.2 実AIの利用制限と費用管理

- 開発中の既定AIは必ず `MockAI` とする。
- Unit、Integration、Leak、E2E、UI確認、デバッグ、CI、反復シミュレーションではOpenAI APIを一切呼ばない。
- OpenAIクライアントの実装とテストには、固定レスポンスを返すMock transportを使う。
- `--ai real` を明示し、かつ環境変数 `ALLOW_REAL_AI=1` がある場合だけ実AIを許可する。片方でもなければ起動を拒否する。
- 実AIは、lint、typecheck、全自動テスト、MockAI 30試合、build、Docker buildがすべて成功した後の**最終受け入れ試験**でのみ使用する。
- 通常の最終受け入れ試験は実AIによる1試合完走だけとする。ただし昼議論の品質を変更した場合は、マスターの明示指示があるときに限り、1日目の処刑までを3試合同時に実行して反復、灰評価、盤面進行、投票予定を検証できる。これは勝率・ゲームバランス測定へ流用しない。
- 実AI試験が失敗した場合は結果と消費呼び出し数を保存する。原因修正後はMockAIで再検証を完了してから、同じ受け入れ範囲を再実行する。
- READMEと`.env.example`にも、実AIは有料であり最終確認専用であることを明記する。

### 4.3 OpenAI SDK

SDKの自動リトライは無効化し、アプリ側だけで回数を管理する。

```ts

```

### 4.4 情報分離

全AIが受け取る情報:

- 一般ルール。
- 自分の座席と役職。
- 全員の座席・生死。
- 公開済み発言、GM発表、記名投票。
- 現在の日・フェーズ・求められている判断。

役職固有:

- 人狼: 仲間、生存人狼だけの夜会話、過去の最終襲撃先と襲撃の成否。
- 占い師: 自分の全占い履歴と結果。
- 霊媒師: 自分の全霊媒結果。
- 狩人: 自分の全護衛履歴と護衛の成否。
- 狂人: 追加情報なし。人狼を推理して支援する。
- 占い師・霊媒師・狂人・人狼: 自分にだけ与えられた役職主張directive。directiveは公開主張として認可された役職・結果だけを含み、真偽を表す語や他者の秘密役職を含めない。

渡してはいけない情報:

- 他人の本当の役職。
- 他役職の非公開結果。
- 未公開票。
- 人狼以外への人狼会話。
- GM視点用の判断メモ。
- OpenAIの内部reasoning。

### 4.6 Structured Outputs

```ts
const SpeechDecision = z.object({
  speech: z.string(),
  addressedTo: z.enum(legalSeatIds).nullable(),
  requestsReply: z.boolean(),
});

const SpeechIntentDecision = z.object({
  urgency: z.union([z.literal(0), z.literal(1), z.literal(2), z.literal(3)]),
  motivation: z.enum(['reply', 'question', 'challenge', 'new_information', 'clarify', 'none']),
  targetSeat: z.enum(legalSeatIds).nullable(),
});

const TargetDecision = z.object({
  targetSeat: z.enum(legalSeatIds),
  statedReason: z.string().max(120),
});
```

- 対象は名前ではなく `seat-1`〜`seat-9`。
- 合法対象のみを動的enumへ入れる。
- `statedReason` は短い説明であり、chain-of-thoughtを要求しない。
- 発言、投票、襲撃、占い、護衛をstrict schemaで処理する。

### 4.7 リトライ

- 1、2、4、8秒の指数バックオフを使い、失敗種別ごとに上限を分ける。
- 接続失敗、タイムアウト、408、409、429、5xxは最大5回。
- refusal、structured output、parse・Zod検証失敗は最大3回。
- ローカルの発言・主張契約違反は具体的な修正指示を付けて最大2回。補正可能な付随メタデータ不一致はAPIを再試行しない。
- 400、401、403、404は即 `paused_error`。
- 種別ごとの上限到達も `paused_error`。
- 再開は同じ `gpt-5.6-luna` で同じ判断を再試行するだけ。

### 4.8 API呼び出し上限

- `MAX_API_CALLS = 240`。
- リトライを含む物理HTTP呼び出し数を数える。
- 呼び出し直前にDBへカウンタと `in_flight` を保存する。
- 上限へ到達する呼び出しは送信しない。

### 4.9 クラッシュ時の境界

- 保存済みAI応答は再起動後に再利用する。
- OpenAI応答受信後・DB保存前にプロセスが落ちた可能性がある `in_flight` は自動再実行しない。
- `ambiguous_ai_call` として停止し、人間が再試行を選ぶ。
- この狭い障害窓では二重課金の可能性はあるが、ゲームイベントは一度だけ確定する。

## 5. 観戦UI

### 5.1 視点

| 情報 | 公開視点 | GM視点 | 終了後 |
|---|---:|---:|---:|
| 発言、投票、死亡 | ○ | ○ | ○ |
| 非公開イベントの実施（内容は伏せる） | ○ | ○ | ○ |
| 本当の役職 | × | ○ | ○ |
| 人狼会話 | × | ○ | ○ |
| 襲撃・占い・霊媒・護衛 | × | ○ | ○ |
| statedReason | × | ○ | ○ |

- 公開データはサーバー側でホワイトリスト射影する。
- 進行中の公開視点では、非公開イベントを固定ラベルだけの `private_action` に射影し、「人狼確認が行われました。」のように実施事実だけを表示する。元のpayload、対象、判定結果、発言本文、理由、audienceは含めない。
- 進行中の公開視点では `statedReason` を含む非公開判断理由をpayloadから除去する。
- 秘密情報をブラウザへ送ってCSSで隠す方式は禁止。
- ライブ中に公開／GM視点を切り替えられる。
- `finished`、`aborted`、`aborted_budget` の終了後は、公開視点でも全役職・人狼会話・夜行動・判断理由を答え合わせとして開示する。

### 5.2 画面

| パス | 内容 |
|---|---|
| `/` | 新規試合開始、seed・速度、試合一覧 |
| `/match/[id]` | ライブ観戦 |
| `/replay/[id]` | 保存イベントのリプレイ |

ホーム:

- `AI人狼` テキストロゴ。
- seed入力。空なら自動生成。
- 速度: ゆっくり3秒、標準1.5秒、最速0秒。
- `AI人狼を開始` ボタン。
- 試合一覧: 日時、seed、状態、勝者、日数。

ライブ:

- ヘッダー: 日数、昼夜・フェーズ、生存数、公開／GM視点、状態。GM視点と終了後は生存人狼数も表示する。
- 中央: 固有名を持つ9人のカード。座席、生死、死亡日と死因、最新発言、直近の投票先、GM視点と終了後の役職バッジ。発言中のキャラクター画像、名前、セリフをカード一覧の上に大きく表示し、pause中も同じ発言を保持する。進行中は発言ステージと3×3の全カードを下部操作バーより上へ常時収め、発言ステージと投票結果パネルは同時に表示しない。
- 右: 公開された役職主張をコンパクトにまとめ、その下に日区切りと重要イベントの視覚的階層を持つ独立スクロールの時系列ログを置く。発言、記名投票、処刑、夜明け、GM秘密、終了後に開示された秘密を区別する。
- 下: pause/resume、速度、中断。pauseは新規AI判断に加え、ブラウザの再生中音声、発言キュー、表示位置を停止し、resumeで同じ音声位置から再開する。pause前から実行中のAI応答は完了を待つが、resumeまでイベント化しない。
- 投票は投票者名、投票先、正規化した得票バーを表示し、GM視点と終了後は判断理由も展開できる。
- 公開視点で非公開の夜処理中や同時投票中も、直前の公開イベントから表示フェーズを安全に導出し、停止と誤認させない。
- `paused_error`ではモデル名、失敗フェーズ、安全に保存したreasonを日本語化した具体的な原因、再試行を表示する。
- 観戦画面内のルールガイドで、9人の配役、勝利条件、昼夜進行、公開／GM視点の違いを確認できる。
- 新規試合は第0夜、1日目の順に開始を画面全面で示す。1日目の演出は第0夜の占い結果確認後、最初の発言より前に表示し、全画面演出中はVOICEと発言表示を待機させる。以後も夜明け、開票、処刑を2.4〜3.6秒の切替演出で示す。夜明けは襲撃犠牲者名または犠牲者なしを大きく表示し、一拍置いてから新しい日数を表示する。通常の連続する演出間には0.6秒、開票から次の演出までには投票結果を読むため5秒の間を設ける。投票開始は全画面演出にせず、画面上のフェーズ表示だけで示す。当日の開票前に前日の投票結果を注目パネルや人物カードへ再表示しない。襲撃犠牲者と処刑者は名前を表示し、それぞれ専用のCC0効果音を鳴らす。効果音は独立してON/OFFと音量調整ができ、設定は再読込後も維持する。再接続・視点切替で過去イベントを重複再生しない。
- 終了時は勝利陣営に加え、陣営別の全配役、各人の生存・処刑・襲撃、個人勝敗をミニエピローグとして表示する。狂人は判定上の人間かつ人狼陣営であることを併記する。

リプレイ:

- ライブと同じコンポーネントを使う。
- イベントseqのシークバー。
- 前後イベント、前後フェーズ、再生、停止、速度。
- AIとゲームエンジンを再実行しない。

イベント封筒:

```ts
interface MatchEvent<T extends string, P> {
  matchId: string;
  seq: number;
  day: number;
  phase: Phase;
  type: T;
  visibility: 'public' | 'private';
  audienceSeats: number[];
  payload: P;
  createdAt: string;
}
```

新規試合の `match_created` は `rules.claims = 'v1'` と `rules.discussion = 'v3'` を持つ。`discussion_speech` は従来の台詞、宛先、返答要求、`claim` に加えて、v3では公開発言の自己分類である `structure` を保存し、変更のない投票予定再宣言を主要貢献から格下げした場合だけ `contributionDemoted=true` を保存する。`claim` は `null` または `claimedRole` と結果配列だけを含む。`stage = 'opening'` はその人物の当日1回目、`stage = 'free'` は2回目以降の発言を表し、全体の固定一巡は表さない。3回目は合意対象本人にだけ許す追加反論である。`rules.discussion = 'v2'` の保存済み試合はv2の発言schema、プロンプト、キュー、payloadを変更せず再開する。`rules.discussion` markerのない旧試合は従来の固定一巡経路で再開する。公開claim ledger、観戦UI、リプレイは保存済みの `discussion_speech.claim` だけから導出し、真役職・真結果・非公開方針を参照しない。v3の `structure` と `contributionDemoted` は当面、公開観戦画面へ表示せず、議論制御と受け入れ評価にだけ使う。

### 7.7 SQLite

テーブル:

- `matches`: id、seed、status、winner、config_json、speed、api_calls、error_json、timestamps。
- `events`: match_id、seq、day、phase、type、visibility、audience、payload、created_at。PK `(match_id, seq)`。
- `ai_calls`: match_id、call_key、request_hash、status (`in_flight|ok|failed`)、response、attempts、OpenAI request ID、timestamps。PK `(match_id, call_key)`。
- `schema_migrations`。

必須:

- WAL。
- foreign keys ON。
- 1判断のAI記録、生成イベント群、match更新を1トランザクションで保存する。
- commit後だけSSE通知する。
- 保存済み`ok`判断は再利用する。
- `(match_id, seq)`と`(match_id, call_key)`で重複を防ぐ。

### 7.8 決定論

- `Math.random()`禁止。
- `rand(seed, purposeKey)` で用途ごとに乱数を導出する。
- 配役シャッフル、昼の最初の話者と指名のない次話者、GM固定文選択に使う。
- AI出力自体は決定論とみなさない。
- 同seed＋MockAIならイベントpayload列を一致させる。
- 役職主張方針と偽結果は `claims/v1/` 名前空間のpurpose keyから純粋導出する。v1で開始した進行中試合の復旧を守るため、v1の確率表と導出ロジックは変更せず、将来の調整は新しいversion markerと実装を追加する。

### 8.6 最終実AI受け入れ試験

以下の試験は、8.1〜8.5、build、Docker buildがすべて成功した後、全工程の最後に1回だけ実行する。

```bash
ALLOW_REAL_AI=1 npm run sim -- --matches 1 --ai real --seed-base 1000
```

合格条件:

- 1試合がクラッシュせず終端する。
- 使用モデルが `gpt-5.6-luna` である。
- 全リクエストが `reasoning.effort=low` である。
- 240回のAPI上限を超えない。
- 秘密情報の漏洩がない。

勝率やゲームバランスを測るために実AIを複数試合実行しない。議論品質の3試合受け入れは `ALLOW_REAL_AI=1` と `npm run accept:day1` の二重条件で、1日目の処刑後に停止し、試合ID、発言、構造化評価、消費呼び出し数を保存する。投票予定が4人以上ある場合は、複数の投票先が残るか、最多予定先に公開された `人狼` 結果があることを候補競合条件とする。3人目の同一予定以後は同じ予定の追加宣言を0件とする。その時点で対象本人が通常の2発言を使い切っていても、投票前に1回だけ追加の反論枠を保証し、対象本人が合意形成後に実際に応答したことを合格条件にする。追加枠は対象本人だけに与え、通常の発言上限は増やさない。後続発言の5分の3以上が対立候補の根拠付き検討、質問・回答、盤面整理、弁明、役職主張のいずれかを増やすことを求める。同じ対立候補を複数ターンで比較しても、単なる勘ではなく反応や発言を更新していれば貢献として数える。5分の3は、旧問題試合の4/7を不合格に保ちつつ、質問へ直接答えて判断を更新する自然な進行を合格にする実AI試験結果から較正した。灰考察は通常盤面で求めるが、役職主張が4人以上かつ公開された `人狼` 結果がある場合は、役職候補と結果先の比較を優先する標準進行を許す。発言時点の投票予定と最終票の一致率は観測値として保存するが、後続情報を踏まえた独立再評価を妨げるため合否条件にはしない。同一話者の再宣言格下げ数も観測値として報告するが、それ自体を合否条件にしない。問題があればまず保存ログとMockAIで調査し、配役・ルール変更はマスターの承認なしに行わない。
